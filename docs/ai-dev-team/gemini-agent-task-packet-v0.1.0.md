# Gemini Agent タスクパケット v0.1.0

## このドキュメントの目的

Gemini に下読み・ログ要約・GCP 観点レビューを依頼するための標準パケット形式を定義する。
Gemini に渡してよいもの・渡してはいけないもの・出力形式・最終判断ルールを明確にする。

---

## Gemini に渡す情報（タスクパケット形式）

以下の形式でタスクパケットを作り、Gemini に渡す：

```
# Gemini タスクパケット

## タスク種別
[下読み / ログ要約 / GCP観点レビュー / diff一次レビュー / コスト見積もり]

## 背景・目的
[なぜこの作業をするか]

## 読んでほしい資料
[ファイルパス / 内容を直接貼る（秘密情報は除く）]

## 確認してほしい観点
[確認事項を箇条書き]

## 出力フォーマット
[下記の出力フォーマットに従う]

## 重要な制約
- 最終的なアーキテクチャ判断はしない
- .env / Secret Manager の値は含めない
- commit / push / deploy の指示はしない
- 「〜してください」ではなく「〜の点が気になります」という形で報告する
```

---

## Gemini に読ませてよいもの

| カテゴリ | 内容 |
|---|---|
| ドキュメント | `docs/` 配下のマークダウン |
| コード | ソースコード（`bot.js` / `tools/` / `smoke/` 等） |
| 設定ファイル | `package.json` / `Dockerfile` / `.dockerignore` |
| ログ | Cloud Run ログ / GitHub Actions ログ（秘密情報を除く） |
| diff | `git diff` の出力 |
| GCP 設定概要 | Cloud Run サービス名・リージョン・スケール設定（値は除く） |

---

## Gemini に読ませてはいけないもの

| カテゴリ | 内容 |
|---|---|
| 秘密情報 | `.env` / `.env.*` の内容 |
| Secret Manager の値 | API キー・トークン・パスワード |
| 個人情報 | ユーザーデータ・PII |
| 本番データ | Firestore / GCS の本番データ |
| 認証情報 | サービスアカウントキー |

---

## 出力フォーマット

Gemini への出力形式の指示（タスクパケットに含める）：

```markdown
以下の形式で回答してください。

## 要約
[3行以内で要点を述べる]

## 確認した観点と結果
| 観点 | 結果 | 備考 |
|---|---|---|
| [確認項目] | OK / 要確認 / NG | [詳細] |

## 気になった点
[必要があれば箇条書き]

## 最終判断はしない
最終的な採否・実行判断はこさめ PM / じゅんやさんが行います。
```

---

## 最終判断を Gemini にさせないルール

Gemini の役割は **一次処理・要約・観点整理** のみ。
以下は必ず **こさめ PM またはじゅんやさん** が判断する：

- アーキテクチャの採否
- commit / push / deploy の実行
- Secret Manager の値の変更
- 外部 API キーの使用

Gemini の出力は「参考情報」として扱い、そのまま実行しない。

## Cloud Run 観点のチェックリスト

Gemini に Cloud Run 関連のレビューを依頼する際に確認させる観点：

```
## Cloud Run チェックリスト

- [ ] PORT 環境変数を正しく使っているか（EXPOSE 8080 / process.env.PORT）
- [ ] `npm ci --omit=dev` で本番依存のみインストールしているか
- [ ] ヘルスチェックエンドポイントが実装されているか
- [ ] コンテナ起動時間が許容範囲内か
- [ ] メモリ / CPU の設定が適切か
- [ ] 秘密情報が Secret Manager 経由で取得されているか（.env に含まれていないか）
- [ ] ログが Cloud Logging に出力されているか
```

---

## Secret Manager 観点のチェックリスト

```
## Secret Manager チェックリスト

- [ ] 秘密情報が .env ではなく Secret Manager に保管されているか
- [ ] IAM ロールが最小権限で設定されているか
- [ ] シークレットにバージョン管理がされているか
- [ ] アクセスログが有効になっているか
- [ ] プロジェクトごとにシークレットが分離されているか
```

---

## GCS / Firestore 観点のチェックリスト

```
## GCS / Firestore チェックリスト

- [ ] バケット / コレクションがプロジェクトごとに分離されているか
- [ ] アクセス制御（IAM / セキュリティルール）が適切か
- [ ] 本番データと開発データが混在していないか
- [ ] バックアップ設定が適切か（GCS: バージョニング / Firestore: エクスポート）
```

```
# Gemini タスクパケット

## タスク種別

## 背景・目的

次の作業を始める前に全体を把握したい。

## 確認してほしい観点
- 各章の要点を1〜2行で要約する
- ANESTY Board v87.0.x と KOSAME Dev Orchestra v0.1.x が混在していないか確認する
- Human Approval が必要な操作が適切に定義されているか確認する

## 出力フォーマット
[上記の出力フォーマットに従う]

## 重要な制約
- 最終的なアーキテクチャ判断はしない
- .env / Secret Manager の値は含めない
```
