# Kosame Approval Delegation Policy v2.1.0

## 概要

こさめ副社長が推奨YES/NOまで判断してよい範囲と、
じゅんやさん社長の最終YESに残す範囲を分離する。

---

## こさめが判断してよいもの（推奨YES/NOを出せる）

| 判断項目 | こさめの推奨 |
|---|---|
| GeminiにタスクをあてるかClaudeにあてるか | routing推奨 |
| verifyへ進むか（実装後） | proceed推奨 |
| Claude補修に切り替えるか | repair推奨 |
| commit候補かどうか | commit推奨 |
| 通常docs/tools/smoke更新を進めるか | approve推奨 |
| Gemini停止時にClaudeへ代行させるか | fallback推奨 |
| Approval Packetをじゅんやさんに提出するか | submit推奨 |

**こさめの推奨はじゅんやさんへの事前整理であり、最終実行権限ではない。**

---

## じゅんやさんの最終YESが必要なもの

| 操作 | 理由 |
|---|---|
| `git push` | リモートへの公開。取り消しが困難 |
| `git tag` | バージョンタグの発行。対外的な記録 |
| `deploy` | 本番・ステージング環境への反映 |
| Cloud Run 本番反映 | サービス停止リスクあり |
| Secret / .env / API key 変更 | 認証情報の変更は高リスク |
| 課金API実行 | コストの発生 |
| 顧客データ処理 | プライバシー・コンプライアンス |
| 破壊的削除（rm -rf等） | 不可逆操作 |
| 本番環境変更 | 影響範囲が広い |

---

## 委任の原則

```
こさめ → "推奨YES/NO + 理由 + 残リスク" を整理
じゅんやさん → 最終操作のYES/NOのみ判断
```

これにより「YES地獄」を排除し、じゅんやさんが判断に集中できる構造を維持する。
