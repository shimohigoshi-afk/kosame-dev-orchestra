# YES地獄削減ガイド v0.4.3

Claude Code と人間の協力において、過剰な承認（YES地獄）を回避し、生産性を最大化するためのガイドライン。

---

## YES地獄とは

AI が安全な操作（ファイルの読み取りや軽微な編集など）のたびに承認を求め、人間が「YES」を連打するだけの状態。
これは人間の認知負荷を高め、重要な意思決定における注意力を削ぐ原因となる。

---

## 削減のためのベストプラクティス

### 1. 自律実行範囲の最大化
`claude-code-approval-policy-v0.4.3.md` で定義された **Allowed (自律許可)** カテゴリの操作については、確認を求めずに実行する。

### 2. Command Batching の活用
複数の編集や検証を一つの「バッチ」として扱い、最終的な結果のみを報告する。
- 悪い例: 「Aを直していいですか？」「(YES)」「Bを直していいですか？」「(YES)」「テストしていいですか？」「(YES)」
- 良い例: 「AとBを修正し、テストを実行しました。結果は以下の通りです。」

### 3. 「事後報告」へのシフト
低リスクな操作は「許可を得てから実行」ではなく「実行してから報告」にシフトする。
ただし、変更内容は常に `git diff` で確認可能な状態に保つこと。

### 4. 承認の「粒度」の最適化
「この1行を変えていいか」ではなく「このタスク（例: 新機能の追加、バグ修正）を完遂していいか」という粒度で承認を得る。

---

## 人間側（じゅんやさん）の役割

- **方針の承認**: 作業開始時のプラン提示に対し、大枠の方向性が正しいかを判断する。
- **クリティカル・ゲートの守護**: `git commit`, `git push`, `deploy` などの重要局面でのみ、集中して内容を確認する。
- **例外への対応**: AI が判断に迷った場合や、想定外の事態が発生した場合の指示。
